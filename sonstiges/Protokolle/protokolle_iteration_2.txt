Vorstellung neuer Stories
  Story 2
    Es soll ueberprueft werden, ob die Anzahl der Argumente gueltig ist,
    oder ob mehr/ weniger Argumente, als in der Definition verwendet werden.

    Idee zur Umsetzung

      - #Parameter ohne default-values ist Minimum
      - #aller Parameter ist Maximum

      -> UserCodeParser muss auch unbenannte Argumente herausfiltern
      -> Comparator braucht eine Methode, um das zu vergleichen
      -> neue Fehlerausgabe

    Bearbeitung in 2. Iteration


  Story 3
    Werte sollen mit den default-values abgeglichen werden. Falls einem
    Parameter der Defaultwert zugewiesen wird, soll mittels Warning darauf
    hingewiesen werden.

    Umsetzung
      steht noch aus

Freitag 29.5.

    Ablauf
    - Begruessung
    - Kundengespraech (Vorstellung der ersten Story)
    - einfuegen des Feedbacks in Jira (mit Schaetzungen)
    - Selbstaendiges Arbeiten an den Aufgaben
    - Starke Veraenderung der Library Struktur (Hady und Lars)

    Feedback
      - IDE Integration gewuenscht
        -> viel Aufwand, da Einlesen in JavaScript notwendig ist
        -> Umfang einer eigenen Story, siehe Jira
        -> verschoben auf spaeter (Ueberlegung IDE, vs Funktionalitaeten)
      - Kunden war unklar, dass wir den Code statisch analysieren
      - zu pruefendes Modul soll nicht angegeben werden. es sollen immer beide
        geprueft werden
      - Einschraenkungen waren unvollstaendig

      - bei eventuellem Themenwechsel soll der Code entsprechend gut kommen-
        tiert sein, damit sich auch Leute, die den code nicht geschrieben haben
        darin zurecht finden und ihn weiter bearbeiten koennen

      Umsetzung
        - IDE Integrierung wurde als Story in Jira vorformuliert
        - Modulangaben, sollen weg fallen -> Aenderung in code_analyser
        - Protokolle von Treffen genauer machen und auf git laden, damit
          Man einen besseren Eindruck der aktuellen Situation und den
          getroffenen/ abgemachten Einschraenkungen hat

      Sonstiges
        Git
          - Im Develop-Branch werden nur Zwischenversionen gesammelt
          - fuer die Bearbeitung sind eigene Unterbranches zu erstellen
          - darin sollen sich auch die entsprechenden Unittests befinden
          - der Code ist so zu schreiben/ integrieren, dass sich alles
            nach mergen in Develop Problemfrei ausfuehren laesst
